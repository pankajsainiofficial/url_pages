API Docs And Usage

Production-oriented HRMS API guide for frontend integration.

Covers Redis-backed sessions, email verification before account creation, Google OAuth, scaffold resource routes, and frontend request examples.

Base URLs

Backend API http://localhost:4000/api
Health http://localhost:4000/health
Frontend http://localhost:5173

The backend uses JSON for normal API responses and HTTP-only cookies for authentication sessions.

Frontend Fetch Helper

Use credentials: "include" for every authenticated request. This allows the browser to send the Redis-backed session cookie. 

const API_BASE_URL =
  import.meta.env.VITE_API_BASE_URL ?? "http://localhost:4000/api";

export async function apiRequest<T>(
  path: string,
  options: RequestInit = {},
): Promise<T> {
  const response = await fetch(`${API_BASE_URL}${path}`, {
    ...options,
    credentials: "include",
    headers: {
      "Content-Type": "application/json",
      ...options.headers,
    },
  });

  const payload = await response.json().catch(() => null);

  if (!response.ok) {
    throw new Error(payload?.message ?? "API request failed");
  }

  return payload;
}

Authentication

Authentication is server-session based. Sessions live in Redis and the browser receives an HTTP-only cookie named by SESSION_NAME.

Setting Behavior
Default lifetime 8 hours
Remember me lifetime 30 days
Cookie HTTP-only session cookie
Frontend requirement credentials: "include"

Register

POST /api/auth/register

Registration does not create the database account immediately. It stores a short-lived pending registration in Redis, sends an email verification link, and returns 202 Accepted. The organization and user rows are created only after verification.

Request

{
  "organizationName": "Acme HR",
  "organizationEmail": "admin@acme.com",
  "password": "strong-password",
  "termsAccepted": true,
  "rememberMe": false
}

Frontend Usage

await apiRequest("/auth/register", {
  method: "POST",
  body: JSON.stringify({
    organizationName,
    organizationEmail,
    password,
    termsAccepted: true,
    rememberMe: false,
  }),
});

Response

{
  "success": true,
  "message": "Verification email sent. Please verify your email to complete registration.",
  "data": {
    "email": "admin@acme.com",
    "expiresInMinutes": 30
  }
}

Verify Email

GET /api/auth/verify-email?token=<verification-token>

The verification link is sent to the registering user's email.

  • Backend creates the organization and owner user.
  • Backend deletes the pending registration from Redis.
  • Backend creates the same HTTP-only session cookie used by login.
  • Backend redirects to FRONTEND_EMAIL_VERIFIED_URL when configured.

Google Login

GET /api/auth/google

GET /api/auth/google/callback

Frontend Usage

window.location.href = "http://localhost:4000/api/auth/google";

With Remember Me

window.location.href = "http://localhost:4000/api/auth/google?rememberMe=true";

Flow

  1. Frontend redirects to /api/auth/google.
  2. Backend stores OAuth state in the Redis session.
  3. Google redirects to /api/auth/google/callback.
  4. Backend validates state, exchanges the OAuth code, verifies the email, and creates the app session.
  5. Backend redirects to FRONTEND_AUTH_SUCCESS_URL.

Session APIs

Current Session

GET /api/auth/me 

const session = await apiRequest<{ data: { user: unknown } }>("/auth/me");

Refresh Session

POST /api/auth/refresh

This currently returns the active session user and requires an existing session.

Logout

POST /api/auth/logout 

await apiRequest("/auth/logout", {
  method: "POST",
});
